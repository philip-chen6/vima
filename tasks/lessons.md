# lessons

- when the user asks for svg/tab treatment in a sidebar, treat "tabs" literally: enhance the navigation items with per-tab svg icons and labels, not the destination section headers.
- when the user asks for a useful shippable artifact and the scope is within reach, ship the artifact instead of stopping at a strategic plan. research should feed implementation, not replace it.
- if a scroll experiment is called fucked, purge the behavior completely instead of iterating around it: remove pinning attrs, ScrollTrigger pin blocks, css sticky fallbacks, and unused route hooks in one pass.
- hero scroll cues should stay brutally simple when the user calls out clutter: use one lowercase label plus a directional icon, not event metadata plus explanatory scroll copy.
- when the user asks to implement a scroll reference across the website, apply it to the existing page/section structure unless they explicitly ask for a new section. don't add a new conceptual section just to showcase the effect.
- don't bind wheel/touch gesture observers to content sections unless the user explicitly asks for scroll-driven interaction. for proof/evidence sections, scrolling should move the page; tab changes should come from direct controls or keyboard only.
- don't combine global `scroll-behavior: smooth` with GSAP ScrollSmoother anchor navigation. native smooth scroll makes the real scroll position lag behind the smoother transform, causing snap-back/rebound. use one scroll authority and prefer `smoother.scrollTo()`.
- with ScrollSmoother navbar jumps, avoid long double-eased scroll tweens while section reveals and decorative media are active. mark programmatic scrolls, pause/defer expensive work during the tween, and use a shorter out-ease so arrival does not feel frozen.
- when applying the branding rule, preserve the deliberate spaced lowercase `v i m a.` primary logo wordmark. use `vima` for prose/metadata and avoid `VIMA` or all-caps styling unless it is a canonical acronym.
- when applying serif typography in this project, use Times New Roman only on the `v i m a.` wordmark. all subtext, prose, labels, and UI should stay in Saans/Saans Mono unless the user explicitly changes that.
- lorem ipsum placeholder copy is acceptable on the landing page while the visual system is being tuned. don't replace it with product copy unless asked.
- shader background experiments must be isolated behind a graceful fallback and verified against browser console errors before calling them done. never let a decorative WebGL layer crash the landing page.
- when the user asks for a "crazy" shader/background, do not bury it under a heavy readability wash. start with the effect visibly present, then tune contrast only enough for the copy to survive.
- when adding follow-up sections like footers, preserve the existing final cta treatment unless explicitly asked to simplify it. do not accidentally replace the hex shader cta with plain links while editing nearby layout.
- don't leave naked telemetry numbers on ctas. if a metric appears inside a button, it needs a label or enough visual context to read as data instead of random ui debris.
- when the user asks for weighted or heavy scrolling, do not interpret that as scroll snapping. make wheel movement feel damped/inertial while preserving free section positions.
- when the user asks to use a generated video/background, do not silently layer the original still as the visible background. ask or keep it video-only; using the source still changes the direction even if it improves crispness.
- don't gate landing video visibility only on media events like `onCanPlay`; those can be missed around hydration and leave the background invisible. make the video/poster visible by default and use overlays/fallbacks for polish.
- for this landing page, restraint beats component density. don't stack the loader, telemetry rail, graph, matrix, shader cta, hex cta, and oversized footer at once; pick one hero flourish and keep the proof sections mostly quiet.
- landing loaders should be dark, restrained, and deterministic. avoid busy terminal panels, glow-heavy multi-element sequences, and resource/load-event gates; use a short fixed intro with low-motion signal elements so the page never feels laggy or flashy.
- when adding gsap to the landing page, avoid generic fade-only batches. use named timeline labels and component-specific motion vectors, then verify with screenshots that no loader/overlay is hiding the actual page animation.
- for navigation, do not overfit the terminal aesthetic into boxed metadata rails. start from the React Bits Pro docs patterns directly, keep top nav simple, and reserve dense instrument styling for proof surfaces.
- when the user says they cannot see an animation, verify the component is mounted in the route and visible in served html before tuning the animation itself. a component file existing in `components/` does not mean the page renders it.
- when adding smooth anchor navigation with `ScrollSmoother`, keep nav active state in the same GSAP/ScrollTrigger system and lock it during programmatic scrolls. mixing IntersectionObserver with smoother-driven scroll causes highlight lag and flicker.
- when the user asks for dropdown nav animation, build an actual dropdown/mega-menu with navigable rows and a panel under the nav, not a tiny tooltip/preview popover.
- loader-to-hero transitions should reveal the actual hero composition, not a detached duplicate scene. if an expanding mask contains imagery, viewport-lock the image layer so the handoff lands on the same background position as the page underneath.
- with ScrollSmoother, never use scroll `fromTo()` reveals that set visible content back to `autoAlpha: 0` on enter. pre-initialize reveal states during setup, then animate to visible, or the section will flicker while scrolling.
- when using MotionPathPlugin with ScrollSmoother, keep decorative path followers outside `#smooth-content` in their own fixed html wrapper. don't rely on `position: fixed` directly on the outer svg; chrome may treat it like normal svg content.
- when the user asks for a motionpath follower, don't show the actual path unless they explicitly want a visible trace. the path is usually scaffolding; the moving object is the feature.
- if a decorative scroll follower still feels bad after multiple tweaks, remove it completely instead of continuing to polish the distraction. keep the landing page restrained and let the core sections breathe.
- don't infer teammate github handles from first names or vague collaborator names. if a handle isn't explicit in repo context, leave it out or ask; wrong personal links are worse than missing links.
- footer resource areas should not duplicate the main nav or in-page section anchors. keep them to off-site links and concrete hackathon artifacts, or the footer starts feeling cramped and noisy.
- footer logo marks should live inside the same wordmark heading row when the user asks for them inline. a separate flex wrapper can still look visually detached even when it is technically beside the text.
- footer media bands need enough vertical depth to read as video, not a decorative slit. if the band gets taller, move the fade seam and bounce overlay with it so the composition stays coherent.
- footer github chips should prioritize scanability over packing. render name and handle as two stacked lines inside one clickable chip instead of forcing both into a single cramped row.
- for footer image-to-video loops, treat camera motion as a failure unless the user explicitly asks for it. use a locked tripod prompt, strong negative prompt for zoom/pan/dolly, and identical first/last frames when possible so motion stays inside the scene.
- when fixing a gsap component that "exists but doesn't work", verify both mount and mutation in the browser. a rendered client component can still be visually static if the tween target is wrong or the plugin path does not mutate attrs.
- don't reuse the same interaction composition across adjacent landing sections. if proof already uses tabs or left-controls/right-panel, make pipeline a different spatial metaphor, such as a route map, timeline, or animated system diagram.
- with ScrollSmoother and hash jumps, scroll-triggered svg draw animations can miss their initial trigger. keep critical paths visible by default and use GSAP redraw as progressive enhancement instead of leaving the route hidden behind dashoffset.
- when the user says “hexagonal cta animation,” confirm whether they mean the small css hex icon or the background shader before tuning timings. in this landing page, the visually dominant cta animation is the `HeroShader` speed prop.
- section dividers on this landing page need to read as intentional instrument-rule bands, not faint 1px whispers. use enough height, contrast, and label structure that the page clearly separates into scenes.
- osmo/willem-style loaders use a nested growing-image mask, not a viewport-sized image synced behind a slit. stack the images in the small box, fade the extra covers almost instantly, then expand the final image wrapper to viewport.
- when the user says to use loader imagery/site atmospherics across the page, apply that treatment to existing sections as ambient backplates or transitions. do not create new conceptual sections or treat the imagery as isolated loader-only decoration.
- for loader-to-hero handoffs, the final expansion must become the actual full-bleed hero video layer, not a poster inside a letter-slot mask. animate both width and height to viewport and remove borders/shadows before release.
- the hero background layer must be visible from frame zero if the loader reveal uses a transparent aperture. do not hide `[data-gsap-intro="intro-bg"]` behind the intro timeline, or the aperture reveals an empty page.
- when an aperture reveal shows the real background, frame selection still matters. start the shared backdrop zoomed/panned onto an interesting region, then transition it to normal during expansion; otherwise a correct transparent reveal can still look empty.
- loader handoffs should never fade the final image before the aperture fill is transparent and the real backdrop is already moving underneath. keep the last frame alive through the expansion, then land the shared backdrop at the same transform the post-loader page uses.
- when a loader expands into a video hero, the final loader frame needs to be the hero video's own poster/first-frame composition, not a nearby generated still. otherwise the transition technically works but visually lands on the wrong image.
- if a poster-to-video handoff must look seamless, do not autoplay the background video underneath the loader. hold it on frame 0/poster until the loader release event, then start playback after the mask has already matched the frame.
- do not include the shared hero video/backdrop in generic post-loader gsap background loops. the loader owns the reveal; any later transform on `[data-gsap-bg]` reads as a second video sliding or swiping in.
- do not tween the real landing backdrop's x/y during the loader aperture handoff. if the backdrop starts panned/zoomed and moves to final under a fading poster, it looks like a second video swiping in from the side.
- the smoother version is even simpler: use one real background video only and make the loader a transparent aperture/mask over it. duplicated loader videos or poster sync always risk visible handoff drift.
- if the loader video must expand into the hero, keep the loader video full-viewport from the start and animate a clip-path from the aperture rect to full screen. resizing a tiny object-fit video causes crop drift and never lands in the same spot.
- when the user says the entire background should be inside the mini frame, scale the full-viewport background layer down into the aperture instead of clipping a viewport-sized layer through it. add a small bleed around the aperture to avoid subpixel edge leaks.
- if pngs need to cycle before the video reveal, treat them as overfilled cover plates above the scaled mini-background layer. fade the covers away, then reveal the already-fitted video layer under them.
- do not leave a visible aperture border around loader media unless the user explicitly wants a picture-frame. on this loader, the media should sit flush in the word gap with no outline.
- avoid loader first-paint flicker by using `useLayoutEffect` for gsap initial states. do not hide animated letters with a persistent css transform rule, because it can fight gsap and leave the wordmark offscreen.
- when the user asks for gsap timeline reveals to feel slower but keep timing between beats, preserve label positions and offsets. increase tween durations only, not the label schedule.
- loader frame skips usually come from hard-cut fades, timeline speed-ups, or decoding video while image covers are animating. use longer crossfades, avoid `timeScale` acceleration, and only start video playback when it is about to reveal.
- the landing backdrop video should be scoped to the hero. use a ScrollTrigger on `#top` to fade `[data-landing-backdrop]` out after the hero leaves so proof/ledger sections return to black.
- for scroll-dependent backdrop visibility, use a scrubbed opacity tween instead of enter/leave callbacks. callbacks feel like sudden blackouts because the fade begins only after the section boundary is crossed.
- loader logo overlays should use the existing brand mark as a persistent object over the image cycle. rotate it at each cover transition, then tween it toward `[data-gsap-intro="intro-logo-mark"]` so the hero metallic mark can take over.
- when loader logo motion feels too literal, prefer svg path morphing over rotation. morph abstract construction/sakura shapes into the actual mark with `MorphSVGPlugin`, then hand off to the hero metallic logo.
- for loader-to-metallic logo handoffs, do not fade the loader svg before the hero shader exists. reveal the metallic mark underneath first, target the hero mark's untransformed layout rect, then let the loader shell fade as one overlay.
- when the user calls out invented visual assets, stop hand-drawing and research source libraries/generators with clear licensing. for loader morphs, source filled svg paths from open libraries like Tabler and preserve their native coordinate system.
- if gsap loader animation feels laggy, look for simultaneous decode + morph + layout work. pause video playback during heavy morph/expansion windows, slow/space morphs, and start media playback only after loader release.
- post-loader hero intro timelines must wait for a loader-complete signal, not loader-release or timeout fallbacks. release can happen while the loader is still fading; complete means the user can actually see the sequence.
- when the user asks for more apparent separation, do not assume they want a whole highlighted divider band. try one stronger rule first; escalate only if they ask for labels or section chrome.
- before adding gsap to a non-landing surface, inspect the landing gsap implementation first and reuse its motion language. for vima that means scoped `gsap.context`/`useGSAP`, `power3.out`/`expo.out`, subtle transform reveals, scanline/clip style, and no first-paint blanking.
